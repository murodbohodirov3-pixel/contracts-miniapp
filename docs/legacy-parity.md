# Матрица паритета legacy-прототипа

Источник: `legacy/index.html`, зафиксированный в baseline-коммите `279440f`.

| Область | Поведение legacy | Целевой результат и проверка |
| --- | --- | --- |
| Вход | Email/пароль через Supabase, восстановление сессии, выход | Локальный вход через защищённую cookie-сессию; отключение пользователя отзывает сессии. |
| Роли | Администратор видит все договоры и формы, пользователь — вкладки подразделений | API проверяет роль и список разрешённых подразделений; UI не является границей доступа. |
| Подразделения | Список подразделений, вкладка каждого и «Все проекты» | CRUD для администратора и выборка только разрешённых подразделений для пользователя. |
| Договоры | Создание, изменение, удаление, статусы, даты, ИНН, UZS и курс | Создание, редактирование с версией и архивирование вместо физического удаления; фильтры и карточка договора. |
| Файлы | Загрузка в Supabase Storage и временная подписанная ссылка | Частное файловое хранилище в Docker volume, проверка права до скачивания, логическое удаление и аудит. |
| Оплаты | Добавление, изменение и удаление оплат | Добавление, изменение с версией и аннулирование без потери истории. |
| Расчёты | Суммы договора, оплат и остатка в UZS/USD; USD считается по курсу каждой записи | Серверные агрегаты повторяют формулу legacy: `UZS / rate`; parity-тесты на общем наборе данных. |
| Поиск и фильтры | Полнотекстовый поиск, даты, статус и «только с остатком» | Параметризованные API-фильтры и те же фильтры в UI. |
| Отчёты | Excel с листами «Договоры» и «Оплаты», CSV fallback | Серверный XLSX-экспорт с теми же основными столбцами и ограничением по правам. |
| Telegram Mini App | `Telegram.WebApp.ready/expand` и открытие ссылки через Telegram | Один HTTPS-домен, инициализация Mini App; серверная проверка Telegram init data в production. |
| Аудит | Отсутствует | Неизменяемая запись входа и изменений пользователей, договоров, оплат и файлов. |

## Явные изменения относительно legacy

- Supabase Auth, таблицы и Storage заменяются собственными API, PostgreSQL и частным volume.
- Физическое удаление договоров и оплат заменяется архивированием/аннулированием.
- Все критичные проверки прав, расчёты и экспорт выполняются на сервере.
- Номера договоров, лимиты файлов, матрица ролей, домен и правила backup требуют подтверждения владельца данных или IT-команды до соответствующих production-фаз.

## Критерии паритета

1. На одинаковом контрольном наборе совпадают UZS/USD договоров, оплат и остатков.
2. Роль без доступа к подразделению не получает его договоры, файлы, агрегаты или экспорт даже при прямом API-запросе.
3. Файл и данные переживают перезапуск Docker Compose.
4. Пользовательские сценарии: вход, создание/редактирование/архивирование договора, загрузка/скачивание файла, добавление/аннулирование оплаты и экспорт проходят e2e-проверку.
