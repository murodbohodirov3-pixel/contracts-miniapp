# План превращения Contracts Mini App в полноценное приложение

## 1. Цель проекта

Создать внутреннюю систему университета для учета договоров и оплат, которую можно стабильно использовать в браузере и как Telegram Mini App.

За основу берем дизайн, тексты, расчеты и пользовательские сценарии из `legacy/index.html`. Legacy-файл не является целевой архитектурой и после переноса функций не должен участвовать в рабочем приложении.

Целевая система должна:

- хранить пользователей, подразделения, договоры и оплаты в локальном PostgreSQL на сервере университета;
- хранить файлы договоров на диске сервера в постоянном Docker volume;
- не зависеть от Supabase;
- работать после перезапуска сервера и контейнеров;
- быть доступной через домен и HTTPS reverse proxy, который предоставляет IT-команда;
- иметь управляемые миграции, резервные копии, журнал действий и понятный процесс обновления;
- позволять масштабировать frontend, API и фоновые задачи без переделки базы с нуля.

## 2. Текущее состояние

Сейчас проект состоит в основном из одного статического HTML-файла. В нем уже реализованы:

- вход по email и паролю через Supabase Auth;
- роли пользователя и администратора;
- список подразделений;
- создание, изменение и удаление договоров;
- создание, изменение и удаление оплат;
- загрузка и открытие файлов договоров;
- фильтры, поиск, статусы и карточка договора;
- расчеты сумм, оплат и остатков в UZS и USD;
- экспорт договоров и оплат в Excel;
- базовая адаптация под Telegram Web App и мобильный экран.

Чего пока нет:

- собственного backend API;
- локальной базы данных и схемы миграций;
- собственной авторизации и серверной проверки прав;
- Docker-окружения;
- надежного файлового хранилища;
- аудита действий;
- автоматических резервных копий и проверенного восстановления;
- тестов, мониторинга, health checks и процедуры выпуска;
- корректно подключенного локального Git-репозитория с историей GitHub.

## 4. Работа за reverse proxy IT-команды

Сервер не обязан иметь публичный IP. IT-команда направляет публичный домен на внутренний адрес и порт приложения.

Наше приложение должно:

- слушать `0.0.0.0:8080` на сервере;
- отдавать frontend и API с одного домена, чтобы не требовался CORS;
- использовать маршруты `/api/*` для API и `/files/*` для защищенной выдачи файлов;
- доверять `X-Forwarded-For`, `X-Forwarded-Proto` и `X-Forwarded-Host` только от reverse proxy;
- корректно определять HTTPS по `X-Forwarded-Proto` для secure-cookie;
- иметь `/health/live` и `/health/ready` для проверки контейнера и базы;
- не публиковать PostgreSQL-порт `5432` наружу;
- разрешать вход на `8080` только с адресов reverse proxy, если IT-команда может настроить firewall.

Данные, которые нужно получить у IT-команды до production-запуска:

- внутренний IP сервера;
- публичный домен;
- адреса или подсеть reverse proxy;
- заголовки, которые proxy передает приложению;
- максимально допустимый размер загружаемого файла;
- таймаут запросов;
- кто выпускает и обновляет TLS-сертификат;
- доступные каталоги и лимиты диска для базы, файлов и бэкапов;
- правила firewall и порядок открытия внутреннего порта `8080`.

Reverse proxy должен сохранять исходный `Host`, передавать `X-Forwarded-*`, поддерживать загрузку файлов и направлять весь домен на `http://SERVER_INTERNAL_IP:8080`.

## 5. Конфигурация и секреты

В `.env` не храним бизнес-данные, списки подразделений, роли, лимиты интерфейса, пути экранов или другие обычные настройки.

В production секреты предпочтительно передавать через Docker secrets или файлы, доступные только `root`. `.env` допускается как временный вариант, должен иметь права `600` и никогда не должен попадать в Git.

Только критические секреты:

- пароль владельца PostgreSQL;
- пароль отдельного пользователя БД, от имени которого работает API;
- секрет серверных сессий;
- Telegram bot token, когда будет включена серверная Telegram-авторизация.

Не секретные настройки храним в `config/app.yaml` и коммитим:

- внутренний порт;
- допустимые типы и максимальный размер файлов;
- срок пользовательской сессии;
- часовой пояс отображения;
- параметры пагинации;
- имя приложения и feature flags без секретов.

В репозитории должен быть только `.env.example` с названиями переменных и пустыми значениями.

## 6. Проектирование базы данных

Общие правила:

- первичные ключи UUID создаются сервером;
- даты событий хранятся как `timestamptz` в UTC, бизнес-даты договора и оплаты как `date`;
- суммы хранятся как `numeric`, никогда не как `float`;
- курс фиксируется в записи договора или оплаты и не пересчитывает историю задним числом;
- итоговые суммы оплаты и остатка вычисляются запросом, а не дублируются в таблице;
- важные данные не удаляются физически из интерфейса: используется деактивация, аннулирование или `deleted_at`;
- все изменения договора и оплаты выполняются в транзакции вместе с записью аудита;
- внешние ключи, `NOT NULL`, `CHECK` и уникальные индексы защищают данные независимо от frontend;
- API работает под отдельной PostgreSQL-ролью без прав владельца схемы;
- миграции выполняются отдельной ролью только во время выпуска.

### 6.1. Таблицы первой версии

`users`

- `id uuid primary key`;
- `email citext unique not null`;
- `password_hash text not null`;
- `display_name text not null`;
- `is_active boolean not null default true`;
- `last_login_at timestamptz`;
- `created_at`, `updated_at`, `disabled_at`.

`roles`

- справочник ролей: `admin`, `manager`, `viewer`;
- роли создаются idempotent seed-скриптом.

`user_roles`

- связь пользователей и ролей many-to-many;
- уникальная пара `user_id + role_id`.

`departments`

- `id uuid primary key`;
- стабильный уникальный `code`;
- `name`;
- `is_active`;
- даты создания и изменения.

`user_departments`

- определяет подразделения, доступные конкретному пользователю;
- администратор имеет глобальный доступ по разрешению роли, а не через frontend.

`contracts`

- подразделение, контрагент, ИНН, номер и дата договора;
- статус: `active`, `closed`, `paused`, `problem`;
- сумма в UZS и курс договора;
- комментарий;
- `version integer` для защиты от одновременного перезаписывания;
- автор создания и последнего изменения;
- `created_at`, `updated_at`, `deleted_at`.

Уникальность номера договора задается после подтверждения бизнес-правила: глобально или внутри подразделения/года. До этого номер индексируется, но не получает ошибочное ограничение `unique`.

`payments`

- ссылка на договор;
- дата, сумма UZS и курс конкретной оплаты;
- способ оплаты и комментарий;
- автор создания и изменения;
- `version`, `created_at`, `updated_at`, `voided_at`;
- проверки `amount_uzs > 0` и `exchange_rate > 0`.

Оплаты не удаляются каскадно при удалении договора. Договор с оплатами можно только архивировать, а ошибочную оплату аннулировать с причиной.

`contract_files`

- ссылка на договор;
- оригинальное и внутреннее имя;
- относительный путь в volume;
- MIME-тип, размер и SHA-256;
- автор и дата загрузки;
- `deleted_at`.

Путь с диска никогда не возвращается клиенту напрямую. Скачивание проходит через API с проверкой сессии и доступа к подразделению.

`user_sessions`

- хэш непрозрачного session token;
- пользователь, срок действия, IP/user-agent для аудита;
- дата отзыва и последнего использования.

`audit_logs`

- пользователь, действие, тип и ID сущности;
- время, IP, request ID;
- безопасные JSON-снимки `before` и `after` без паролей, токенов и содержимого файлов.

### 6.2. Индексы

Нужны индексы как минимум на:

- email пользователя;
- код и активность подразделения;
- `contracts.department_id`, дату, статус, ИНН и номер;
- `payments.contract_id` и дату оплаты;
- активные сессии пользователя;
- `audit_logs.entity_type + entity_id` и дату события;
- частичные индексы для строк, где `deleted_at is null` или `voided_at is null`.

Поиск первой версии работает по индексированным полям. `pg_trgm` добавляем отдельной миграцией только когда понадобится быстрый нечёткий поиск.

## 7. Иерархия миграций

Миграции лежат только в `db/migrations` и имеют формат:

```text
000001_extensions.up.sql
000001_extensions.down.sql
000002_identity_and_sessions.up.sql
000002_identity_and_sessions.down.sql
000003_departments_and_access.up.sql
000003_departments_and_access.down.sql
000004_contracts.up.sql
000004_contracts.down.sql
000005_payments.up.sql
000005_payments.down.sql
000006_contract_files.up.sql
000006_contract_files.down.sql
000007_audit_logs.up.sql
000007_audit_logs.down.sql
000008_indexes_and_constraints.up.sql
000008_indexes_and_constraints.down.sql
000009_initial_reference_data.up.sql
000009_initial_reference_data.down.sql
```

Правила миграций:

- примененная миграция никогда не редактируется и не переименовывается;
- каждое изменение схемы получает новый следующий номер;
- одна миграция решает одну понятную задачу;
- `up` обязателен, `down` нужен для локальной разработки, но production автоматически назад не откатывается;
- удаление столбца выполняется минимум в два релиза: сначала код перестает его использовать, потом отдельная миграция удаляет;
- большой backfill данных отделяется от изменения структуры;
- перед production-миграцией создается backup;
- миграцию запускает одноразовый контейнер до запуска новой версии API;
- CI проверяет путь как на пустой БД, так и обновление копии предыдущей схемы;
- seed содержит только справочники и права, но не пользователей и не реальные договоры.

Порядок production-обновления:

1. Проверить свободное место и состояние сервисов.
2. Создать и проверить backup PostgreSQL и файлов.
3. Скачать новую версию образов.
4. Выполнить совместимые миграции отдельным migration job.
5. Запустить API и frontend.
6. Проверить health endpoints и smoke tests.
7. Сохранить логи выпуска и номер примененной миграции.

## 8. Backend API

Минимальные группы маршрутов:

```text
POST   /api/auth/login
POST   /api/auth/logout
GET    /api/auth/me

GET    /api/departments

GET    /api/contracts
POST   /api/contracts
GET    /api/contracts/:id
PATCH  /api/contracts/:id
POST   /api/contracts/:id/archive

GET    /api/contracts/:id/payments
POST   /api/contracts/:id/payments
PATCH  /api/payments/:id
POST   /api/payments/:id/void

POST   /api/contracts/:id/files
GET    /api/files/:id/download
DELETE /api/files/:id

GET    /api/reports/contracts.xlsx
GET    /api/audit
GET    /health/live
GET    /health/ready
```

Требования к API:

- единый формат успешного ответа и ошибки;
- строгая проверка всех входных данных;
- авторизация и права проверяются на сервере для каждого маршрута;
- пагинация и серверные фильтры для списков;
- транзакции для связанных изменений;
- request ID во всех логах и ответах;
- ограничения размера тела запроса и файла;
- rate limit на вход;
- безопасные HTTP-заголовки;
- журналирование без паролей, session token и содержимого договоров;
- корректное завершение активных запросов при перезапуске контейнера.

Авторизация первой версии: email/пароль с Argon2id и серверная сессия в `HttpOnly`, `Secure`, `SameSite=Lax` cookie. Мутации дополнительно проверяют `Origin`. Секретные роли и права не принимаются от клиента.

Telegram Mini App не должен доверять данным пользователя из JavaScript. Если включаем Telegram-вход, API проверяет `initData` с bot token, связывает Telegram ID с существующим пользователем и только после этого создает обычную серверную сессию.

## 9. Файлы договоров

- Файлы хранятся вне контейнера в именованном volume или выделенном каталоге сервера.
- В БД хранится только метаинформация и относительный внутренний путь.
- Имя на диске генерируется сервером и не содержит пользовательский путь.
- Разрешаются только согласованные типы, например PDF, DOCX, XLSX, JPG и PNG.
- Проверяется фактический MIME-тип, расширение, размер и SHA-256.
- Файл сначала пишется во временный каталог, затем атомарно перемещается после успешной записи БД.
- Удаление сначала логическое; физическая очистка выполняется фоновой задачей после срока хранения.
- В backup всегда входят и PostgreSQL, и files volume одного периода.
- При необходимости антивирусной проверки позднее добавляется отдельный worker, не меняя контракт API.

## 10. Frontend и перенос legacy-функций

Новый frontend повторяет существующие рабочие сценарии, но разбивается на поддерживаемые модули:

- страница входа;
- общий layout и навигация по подразделениям;
- dashboard со статистикой;
- список и фильтры договоров;
- создание и редактирование договора;
- карточка договора и файлы;
- оплаты;
- пользователи и права для администратора;
- аудит;
- экспорт отчета.

Расчеты денег должны находиться в общих протестированных функциях. API является источником итоговых данных, а frontend только отображает их. Суммы передаются строками decimal или в согласованном minor-unit формате, чтобы JavaScript не терял точность.

Перенос выполняется вертикальными частями. После переноса каждого сценария он сравнивается с `legacy/index.html` и тестируется целиком через API и PostgreSQL.

## 11. Перенос данных из Supabase

Если в Supabase уже есть реальные данные, отключать его можно только после контролируемого переноса.

Порядок:

1. Зафиксировать фактическую схему, пользователей, RLS и количество строк.
2. Снять полный экспорт таблиц `profiles`, `departments`, `contracts`, `payments`.
3. Скачать все файлы из bucket `contracts` и сверить количество и checksum.
4. Преобразовать ID, роли и пути файлов в новую схему отдельным повторяемым скриптом.
5. Импортировать сначала пользователей и подразделения, затем договоры, оплаты и файлы.
6. Сравнить количество строк, суммы договоров, суммы оплат и остатки по каждому договору.
7. Провести пробный перенос на тестовой БД.
8. На финальном переносе временно остановить запись в старую систему.
9. Повторить выгрузку, импорт и сверку.
10. Переключить домен на новое приложение и оставить Supabase только для чтения на согласованный период.
11. Удалить Supabase-ключи и SDK из рабочего frontend после подтверждения владельца данных.

Пароли Supabase Auth экспортировать в пригодном для входа виде обычно нельзя считать гарантированным сценарием. Для пользователей планируем безопасную установку новых паролей или одноразовые приглашения.

## 12. Docker Compose

Production Compose содержит:

- `gateway`: публикует только `8080`, раздает frontend и направляет `/api` и `/files` в API;
- `api`: не публикует порт на host, подключается только к внутренней сети;
- `postgres`: не публикует `5432`, использует постоянный volume;
- `migrate`: одноразовый профиль/контейнер для миграций;
- `backup`: плановый job или скрипт, запускаемый host timer/cron;
- опциональный `worker` появится позже для тяжелых отчетов, очистки и уведомлений.

Каждый образ имеет фиксированную версию. Не используем `latest`. Контейнеры запускаются от непривилегированного пользователя, имеют health checks, restart policy и лимиты логов.

Постоянные данные:

```text
postgres_data  -> данные PostgreSQL
contract_files -> файлы договоров
backup_data    -> локальные временные backup-копии
```

Backup должен копироваться на отдельный диск или внешний защищенный storage. Backup на том же физическом диске не защищает от поломки сервера.

## 13. Безопасность

- PostgreSQL и API не доступны напрямую из публичной сети.
- SSH доступен только администраторам, предпочтительно по ключам.
- Production-секреты не хранятся в Git и не попадают в логи.
- Пароли хэшируются Argon2id.
- Сессии можно отозвать при отключении пользователя.
- Все права проверяются API, а не скрытием кнопки.
- Файлы нельзя получить по угадываемому пути.
- Вход, изменения прав, договоров, оплат и файлов записываются в аудит.
- Для опасных действий используется подтверждение и причина.
- Dependencies и образы регулярно обновляются отдельным проверенным релизом.
- На reverse proxy включаются HTTPS, ограничение тела запроса и базовые security headers.
- Время сервера синхронизируется через NTP.

## 14. Резервные копии и восстановление

Минимальная политика:

- ежедневный `pg_dump` в custom format;
- ежедневный архив новых/измененных файлов;
- хранение минимум 7 ежедневных и 4 еженедельных копий, затем политика уточняется с университетом;
- шифрование backup, если он покидает сервер;
- checksum и журнал результата каждого backup;
- уведомление при ошибке или отсутствии свежей копии;
- ежемесячное тестовое восстановление в отдельный PostgreSQL-контейнер;
- документированный RPO и RTO после согласования с владельцем системы.

Система не считается защищенной бэкапами, пока не выполнено успешное тестовое восстановление БД и файлов.

## 15. Логи и эксплуатация

- API пишет структурированные JSON-логи в stdout.
- Docker ограничивает размер и число log-файлов.
- Логи содержат время, уровень, request ID, маршрут, статус и длительность.
- Health check различает живой процесс и готовность базы.
- Отдельно отслеживаются ошибки входа, API 5xx, заполнение диска, свежесть backup и недоступность PostgreSQL.
- В `ops/runbooks` создаются инструкции: первый запуск, обновление, rollback приложения, восстановление backup, создание администратора и отключение пользователя.

## 18. Приемочный чек-лист

- Пользователь входит и выходит, cookie защищена и сессия отзывается.
- Администратор создает и отключает пользователей.
- Пользователь видит только разрешенные подразделения.
- Договор создается, редактируется и архивируется.
- Файл загружается, скачивается только с правами и сохраняется после restart.
- Оплата добавляется и аннулируется без потери истории.
- Суммы UZS/USD и остатки совпадают с контрольными расчетами.
- Фильтры и Excel-отчет работают на реальных объемах данных.
- Аудит показывает кто, что и когда изменил.
- PostgreSQL недоступен снаружи.
- Приложение корректно работает за reverse proxy и HTTPS-доменом.
- Telegram Mini App открывает тот же домен и проходит серверную проверку пользователя.
- После рестарта сервера Compose поднимает сервисы автоматически.
- Backup создается, проверяется и успешно восстанавливается.
- Миграции применяются на чистой и предыдущей версии БД.
- В Git и образах нет production-секретов и реальных файлов договоров.

## 19. Решения, которые нужно согласовать до разработки соответствующего этапа

- точный публичный домен и параметры reverse proxy;
- список ролей и матрица прав;
- глобальная или локальная уникальность номера договора;
- список подразделений и кто их администрирует;
- допустимые форматы и максимальный размер файлов;
- срок хранения удаленных файлов, аудита, сессий и backup;
- нужны ли Telegram-вход и привязка Telegram ID в первой production-версии;
- есть ли реальные данные в Supabase и кто подтверждает результаты переноса;
- требуемые RPO/RTO и место внешнего хранения backup;
- staging на этом же сервере или отдельная тестовая машина.
